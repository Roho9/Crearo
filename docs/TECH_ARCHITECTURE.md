# TECH_ARCHITECTURE.md — Crearo iOS

*The engineering plan behind GDD §61. This repo implements the non-3D layers described here; the 3D world is layered on top via RealityKit (see §4).*

---

## 1. Stack at a glance

| Concern | Choice | Why |
|---------|--------|-----|
| Language | **Swift 5.10+** | native, safe, first-class on Apple platforms |
| 2D UI / menus / "making surfaces" | **SwiftUI** | declarative, fast to build, great for the creative input UI |
| 3D world | **RealityKit** (native) | modern, ground-up **replacement for SceneKit**, best fit for "simple lightweight 3D" on Apple HW; integrates with SwiftUI via `RealityView` |
| Architecture | **MVVM + protocol-oriented services + SPM core** | testable, swappable, clean separation |
| Shared logic | **`CrearoCore` Swift Package** | platform-agnostic models/engines, unit-testable without a device |
| Backend | **Supabase** (Postgres + Auth + Storage + Edge Functions + **pgvector**) | the rarity/originality engine needs vectors; Sign in with Apple built-in |
| Private media | **CloudKit** (optional) | cheap, private per-user storage for drawings/photos/voice |
| AI/ML | **server Edge Functions** (LLM/vision/embeddings) + **on-device Core ML/Vision** | keys off-device; cheap private pre-checks on-device |
| Local persistence | **SwiftData** (or GRDB/Core Data) | local-first world-state, offline play |
| Min target | iOS 17+ (RealityView, SwiftData), iPhone 12-class GPU | balances reach with modern APIs |

> RealityKit is the MVP choice. **Unity-as-a-Library** is the documented escape hatch (§4.3) if 3D scope outgrows RealityKit or we later go cross-platform — the architecture isolates the 3D layer so this swap doesn't touch game logic.

## 2. Module / folder layout

```
Crearo/                            ← repo root (Xcode workspace generated by XcodeGen)
├── project.yml                    ← XcodeGen spec → run `xcodegen generate`
├── CrearoCore/                    ← Swift Package: ALL platform-agnostic logic (no UIKit/SwiftUI/RealityKit)
│   ├── Package.swift
│   ├── Sources/CrearoCore/
│   │   ├── Models/                ← Codable domain models (profile, creation, world, economy)
│   │   ├── Engines/               ← CreativityScoringEngine, BalanceEngine, RarityService, BossComposer
│   │   └── Services/              ← protocols + DTOs: AIInterpretation, Networking, Auth, Persistence
│   └── Tests/CrearoCoreTests/     ← pure unit tests (run on Linux/CI without a simulator)
├── CrearoApp/                     ← SwiftUI app target (depends on CrearoCore)
│   ├── CrearoApp.swift            ← @main, composition root
│   ├── AppState.swift             ← top-level observable app state
│   ├── RootView.swift             ← navigation shell
│   ├── DesignSystem/              ← Theme (the color = creativity language)
│   ├── Features/<Feature>/        ← MVVM: <Feature>View.swift + <Feature>ViewModel.swift
│   │   ├── CharacterCreation/  HomeBase/  Forest/  CreationForge/  DailyQuest/  GrowthReport/
│   └── Services/                  ← concrete implementations + DI container (AppServices)
└── supabase/                      ← schema.sql + Edge Functions (interpret-idea, score-originality)
```

**Dependency direction:** `Features → ViewModels → CrearoCore (protocols/engines)`. Concrete service impls live in `CrearoApp/Services` and are injected at the composition root, so `CrearoCore` never depends on a specific backend, network library, or Apple UI framework. This is what makes the data layer swappable and the logic testable.

## 3. MVVM + dependency injection

- **Model:** `Codable` value types in `CrearoCore/Models` (e.g., `Creation`, `CreativeProfile`, `ResourceWallet`). No I/O, no UI.
- **ViewModel:** an `@Observable` (Observation framework) or `ObservableObject` per feature, owning state and calling **service protocols** (never concrete impls). Pure-ish, testable.
- **View:** SwiftUI, dumb, binds to the ViewModel. The 3D `Forest` view hosts a `RealityView`.
- **Services (protocols in `CrearoCore/Services`):** `AuthService`, `PersistenceStore`, `APIClient`, `AIInterpretationService`, `CreativityScoring`, `RarityService`. Concrete impls (`SupabaseAuthService`, `SwiftDataStore`, `LiveAPIClient`, `RemoteAIInterpretationService`, …) live in the app and are wired by `AppServices` (a tiny DI container).
- **Why DI/protocols:** unit tests inject fakes (`InMemoryStore`, `MockAIInterpretation`); we can swap Supabase↔Firebase↔CloudKit by writing a new impl of the same protocol; previews use stubs.

```swift
// composition root (CrearoApp/Services/AppServices.swift) — sketch
let services = AppServices(
    auth: SupabaseAuthService(client: supabase),
    store: SwiftDataStore(),
    api: LiveAPIClient(baseURL: Config.apiBaseURL),
    ai: RemoteAIInterpretationService(api: ...),
    scoring: CreativityScoringEngine(),          // from CrearoCore, pure
    balance: BalanceEngine()                      // from CrearoCore, pure
)
```

## 4. The 3D layer (RealityKit) and the escape hatch

### 4.1 Why RealityKit for the MVP
RealityKit is Apple's modern 3D framework and the **ground-up successor to SceneKit** (SceneKit is no longer getting updates). It hits the "cozy, lightweight, native" target, runs on Metal, integrates with SwiftUI via `RealityView`, and supports ECS, PBR-lite materials, baked lighting, and custom shaders (Shader Graph / `CustomMaterial`) — enough for our stylized look without Unity's overhead. For a small, stylized, Apple-only diorama world, it's the lean choice.

### 4.2 How it slots in
- `Forest/ForestView.swift` hosts a `RealityView`; `ForestSceneController` owns the `Entity` graph (player, props, lights), input mapping (the floating joystick), the camera follow rig, and the **"Grey" corruption material** (a global desaturate/grid post effect applied per-entity, cheap).
- The 3D scene is fed by `CrearoCore` world-state (regions, creations, decay) — the scene is a *view* of game state, so logic stays testable and engine-agnostic.
- Content is a **modular kit** (trees, rocks, walls, props) with LODs and instancing to hit the budget (GDD §8): ≤100k tris, ≤80 draw calls, baked lighting + a few dynamic lights, distance/frustum culling.
- User-created art enters as **restyled decals/billboards/sprites** (the Restyle pipeline, §6), not raw imports, preserving cohesion.

### 4.3 Unity escape hatch
If 3D scope grows (richer animation, larger open spaces, cross-platform), Unity ships as a **library embedded in the native app (Unity as a Library)**, rendered into a view that the SwiftUI shell hosts; `CrearoCore` and the backend are unchanged. We document this now so the architecture never assumes RealityKit internals leak into game logic. Decision gate in `ROADMAP.md`.

## 5. Backend: Supabase vs Firebase vs CloudKit

**The deciding factor:** the originality/rarity engine (GDD §41, scoring §4) needs **vector similarity over a per-prompt population**. That is a SQL/`pgvector` job. It also needs cross-user aggregation, server-side AI brokering, and Sign in with Apple.

| Capability | **Supabase** (recommended) | Firebase | CloudKit |
|------------|---------------------------|----------|----------|
| Vector/rarity (`pgvector` HNSW) | ✅ native, the key reason | ⚠️ no native vectors (needs add-on/Vertex) | ❌ none |
| Relational analytics (rarity baselines, profiles) | ✅ Postgres/SQL | ⚠️ NoSQL, awkward aggregation | ❌ limited querying |
| Auth incl. Sign in with Apple | ✅ | ✅ | ✅ (Apple ID only) |
| Server-side AI brokering | ✅ Edge Functions (Deno/TS) | ✅ Cloud Functions | ⚠️ limited (no general compute) |
| Private per-user media, cheap | ✅ Storage | ✅ Storage | ✅✅ free, very private |
| Cross-platform later | ✅ | ✅ | ❌ Apple-only |
| Cost predictability | ✅ (self-hostable, flat Pro) | ⚠️ usage spikes | ✅ free tiers |
| Apple-native offline/sync | ⚠️ DIY | ⚠️ DIY | ✅✅ best |

**Recommendation — Supabase primary + CloudKit for private media:**
- **Supabase** for accounts/auth (Sign in with Apple), the CreativeProfile, the rarity corpus + `pgvector`, the resource economy ledger, Edge Functions for AI/embeddings (§6), and shared/social data later. Postgres + SQL is the right tool for the analytics this game *is*.
- **CloudKit (optional)** for large *private* user media (drawings/photos/voice) in the user's own iCloud DB — minimizes our storage cost and maximizes privacy (the media never leaves Apple's per-user container; we keep only embeddings/derived features server-side). Great fit for offline + sync of the player's *own* world.
- **Firebase** is the documented alternative if the team prefers Google's ecosystem or wants the fastest possible MVP; the protocol-based data layer (§3) means we can implement `SupabasePersistence` or `FirebasePersistence` without touching features. Its weakness is exactly our core need (vectors/analytics).

Because all data access is behind `PersistenceStore`/`APIClient`/`AuthService` protocols, **the backend is a swappable implementation detail**, not an architectural commitment.

## 6. AI interpretation & embedding pipeline

All model calls are **server-side** (keys never on device) via Supabase **Edge Functions**:

- **`interpret-idea`** (GDD §33): input = idea (text/transcript/image ref) + context (level, region, resources, class lean, profile, constraint). Calls an LLM with a **strict JSON schema** + the game's balance rules in the system prompt; output = `{itemType, traditionalStats, creativeStats, uniqueEffect (from allowlist), artDescriptor, suggestedName}`. The function **validates & clamps** against budget before returning (defense-in-depth; the client `BalanceEngine` also clamps).
- **`score-originality`** (scoring §4): input = response (+prompt). Computes embedding (text/image), runs the `pgvector` k-NN rarity query, returns rarity percentile + semantic-distance, plus an LLM-judge originality estimate during cold-start. Writes the (gate-passing) embedding to `prompt_responses` for the growing baseline.
- **On-device (Core ML / Vision / Speech):** speech-to-text, image segmentation for photo tasks, and cheap **coherence/relevance pre-checks** so obvious nonsense is filtered *before* a paid server call. This both protects privacy and cuts cost.
- **Cost control:** on-device pre-filter → cache identical/similar requests → cheap embedding model → batch rarity → the creation **resource economy caps creation frequency**, which conveniently caps inference spend (GDD §65.6).
- **Restyle service:** user art → cohesive dark-fantasy asset. v1 = deterministic (palette-map to region scheme + outline + world lighting + decal). Later = generative (image model keyed to `artDescriptor`) with the same palette/lighting constraints so output stays on-style. Runs server-side or on-device depending on model.

```
device: capture → on-device pre-check (coherence/STT/segment)
   └─→ Edge: score-originality (embed + pgvector rarity)  ─┐
   └─→ Edge: interpret-idea (LLM + schema + balance clamp) ─┤→ balanced, named, restyled Creation
                                                            └─→ persisted (local + cloud), world updates
```

## 7. Offline, world-state & the decay scheduler

- **Local-first:** the player's world-state (home, creations, profile cache, wallet) lives locally (SwiftData) so the game is fully playable offline; mutations queue and sync when online.
- **Authoritative timestamps:** world-state carries server-synced timestamps. **Decay/corruption (GDD §32/§48) is computed against a server clock**, not device time, so it can't be cheated by changing the clock and is consistent across devices.
- **Decay scheduler:** on app open (and via a periodic server job), compute elapsed-since-last-meaningful-creative-activity and apply graceful, reversible corruption tiers; surface restoration quests. Creative-health (low-effort streaks) also feeds the decay input, per §38.
- **Conflict resolution:** last-writer-wins per creation with a server-side merge for the profile/economy ledger (append-only events → derived state), avoiding lost creations on multi-device.

## 8. Security & privacy (engineering)

- **Secrets:** no API keys in the app binary; all third-party model keys live in Edge Functions / server env. The app holds only the Supabase anon key (RLS-protected).
- **RLS everywhere:** Postgres Row-Level Security so a user can only read/write their own profile, creations, and wallet; the rarity corpus is written via a function that strips identity (embeddings + minimal metadata only).
- **On-device first:** intimate media processed locally where possible; store derived features over raw media; private media in CloudKit's per-user container.
- **Encryption** in transit (TLS) and at rest; full export/delete (GDPR/CCPA); content safety classification before any sharing/pooling; minor protections and age-gating (GDD §62).
- **Moderation hooks** in the pipeline for any content that could become social (§60).

## 9. Testing & CI

- **`CrearoCore` is pure** (Foundation-only) → its tests run on **Linux CI** without a simulator (fast, cheap). Covers the scoring engine, balance clamps, economy, rarity math (with a fake vector source), boss composition.
- **App layer:** XCTest + SwiftUI previews with stubbed services; snapshot tests for key views; a thin set of UI tests for the core loop.
- **CI:** GitHub Actions — `swift build && swift test` for `CrearoCore` on Linux + `xcodebuild test` on a macOS runner for the app; `xcodegen generate` to produce the project; SwiftLint. See `SETUP_GUIDE.md`.
- **Determinism:** the scoring engine is deterministic given inputs (embeddings injected via `RarityService` protocol), so tests assert exact behavior; the LLM steps are mocked at the protocol boundary.
